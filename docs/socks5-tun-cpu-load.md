# Нагрузка CPU в гибридном режиме SOCKS5+TUN при больших закачках

## Контекст

При активном роутинге по приложениям (App Routing) в режиме SOCKS5 и большой по объёму закачке (например, торрент на десятки гигабайт) CPU заметно греется. Вопрос, который разбирался: можно ли убрать TUN из SOCKS5-режима полностью, даже когда роутинг по приложениям включён — и почему трафик неотфильтрованных приложений (например, торрент-клиента, который не выбран для роутинга) всё равно создаёт нагрузку.

Документ фиксирует разбор причин и рассмотренные варианты решения. **Решение пока не принято, изменений в коде не вносилось.**

---

## Почему TUN продолжает работать, даже если выбранные приложения — не torrent-клиент

Когда в `AppRoutingStore` выбрано хотя бы одно приложение, `ConfigTransformer.makeSocks5Config` **не удаляет** `tun`-инбаунд — наоборот, специально его оставляет:

```
ConfigTransformer.swift:40-57
  hybridTun = !routedProcessNames.isEmpty
  если !hybridTun → удалить tun-инбаунд (чистый SOCKS5)
  если hybridTun  → tun-инбаунд остаётся активным
```

TUN на macOS перехватывает **весь** IP-трафик системы, который проходит через основной сетевой интерфейс — не только трафик выбранных приложений. У TUN нет возможности захватывать "только трафик процесса X": он работает на уровне сетевого интерфейса, ниже уровня процессов.

```
Весь системный трафик (включая торрент-клиент)
        │
        ▼
  TUN-интерфейс (kernel)
        │
        ▼
  sing-box (userspace, root)
        │
        ├── find_process: true → process_name == выбранное приложение?
        │         да  → route: proxy
        │         нет → route: direct
        │
        ▼
  обратно в kernel → наружу в сеть
```

**Ключевой момент:** даже когда правило выдаёт `direct` для трафика торрент-клиента, пакет уже совершил круг kernel → userspace sing-box → kernel. Именно этот kernel/userspace хаер-пин для **каждого пакета** всей системы — источник базовой нагрузки гибридного режима, независимо от того, фильтруется что-то "по смыслу" или нет.

Сравнение с чистым SOCKS5 (без аппов в роутинге): `tun`-инбаунд удаляется (ConfigTransformer.swift:46), sing-box поднимает только `mixed`-инбаунд на `127.0.0.1:<port>`. Трафик приложений, которые не настроены явно на этот прокси (в т.ч. торрент-клиент), идёт через обычный сетевой стек ОС и вообще не касается sing-box — 0% нагрузки от sing-box.

## Вторая причина — `stack: "mixed"` на TUN-инбаунде

TUN-инбаунд во всех генерируемых конфигурациях всегда создаётся с одним и тем же значением `stack`, без какой-либо настройки в UI:

```
SubscriptionParser.swift:414-422
  "type":         "tun",
  "tag":          "tun-in",
  "address":      [...],
  "auto_route":   true,
  "strict_route": true,
  "stack":        "mixed"
```

В sing-box `stack` определяет, как обрабатываются перехваченные TUN пакеты:

| stack | Как обрабатывается | Стоимость CPU |
|---|---|---|
| `system` | Системный стек ядра делает основную работу | низкая |
| `gvisor` | Полная реализация TCP/IP в userspace (Go) | высокая |
| `mixed` | Комбинация — обычно TCP через system, UDP через gvisor (выбор зависит от версии sing-box и платформы) | средняя/высокая для UDP |

Торрент-клиенты массово используют **UDP** (протокол uTP) как основной транспорт для передачи данных — это именно тот тип трафика, который в режиме `mixed` чаще всего обрабатывается через тяжёлый userspace-стек gVisor. Это веро­ятный главный вклад в нагрев CPU именно при больших закачках (а не просто факт прохождения через TUN как таковой).

## Почему TUN нельзя убрать полностью, сохранив автоматический роутинг по приложениям

Маршрутизация по `process_name` в sing-box (`find_process: true`, ConfigTransformer.swift:130) требует знать, какой процесс инициировал соединение. На macOS единственный способ получить эту информацию для **любого** процесса — перехватить трафик на уровне TUN-интерфейса до того, как он попадёт в обычный сетевой стек, и сопоставить соединение с процессом через системные вызовы.

Проверено, что приложение **не использует** официальный механизм Apple для per-app VPN:

- В `YurecClient.entitlements` нет `com.apple.developer.networking.networkextension` — соответственно, недоступны `NEAppProxyProvider` / `NEPacketTunnelProvider`.
- sing-box запускается как обычный root-процесс через правило в sudoers (`SudoersManager.swift`), а не как System Extension.

Без этого entitlement (который Apple выдаёт в основном под MDM/enterprise-сценарии) per-app VPN для произвольного стороннего приложения на macOS реализовать нельзя. Альтернатива на уровне `pf` (`divert-to`/`rdr-to`) умеет фильтровать по UID/GID локального сокета, но не по имени процесса/исполняемого файла — непрактично для выборочного роутинга обычных GUI-приложений.

**Итог:** пока приложение не перейдёт на NetworkExtension/per-app VPN (отдельная архитектурная работа, требующая entitlement от Apple), TUN физически необходим для **автоматического** роутинга трафика по процессам на macOS.

## Рассмотренные направления решения (не реализованы)

### Вариант A — тюнинг `stack` TUN-инбаунда

Поменять `"stack": "mixed"` → `"stack": "system"` в `SubscriptionParser.swift:421` (и аналогично в `ConfigTransformer.sanitizeTunInbound`, если профиль импортирован извне с другим значением stack).

- Сохраняет всю текущую функциональность — TUN и авто-роутинг по приложениям продолжают работать как сейчас.
- Низкий риск, минимальное изменение.
- Требует эмпирической проверки: на macOS бывают нюансы совместимости UDP при чистом `system` stack — возможно, именно поэтому сейчас стоит компромиссный `mixed`. Нужно протестировать на реальной торрент-закачке перед тем как менять дефолт.

### Вариант B — убрать TUN из SOCKS5-режима полностью

Никогда не поднимать TUN в SOCKS5-режиме (убрать `hybridTun`-ветку в `ConfigTransformer.makeSocks5Config` и соответствующую логику в `ProxyManager.swift`). Вместо авто-роутинга по `process_name` — пользователь сам указывает SOCKS5/HTTP-прокси (`127.0.0.1:<port>`) в настройках конкретных приложений, которые это поддерживают (Telegram, браузеры и т.п.). Это уже частично работает сегодня: трафик, который приходит непосредственно на `mixed`-инбаунд, всегда уходит в proxy независимо от `process_name`-правил (см. комментарий "Rule 0", ConfigTransformer.swift:119-127, про Telegram).

- Торрент-клиент, если его не настраивать на использование прокси, идёт мимо sing-box полностью — TUN не поднимается вообще, 0% нагрузки от sing-box для его трафика.
- Минус: теряется автоматический форс-роутинг для приложений, у которых нет собственных настроек proxy.
- Требует переделки UI (`AppRoutingStore`, `GeneralTabView.swift`, `ProfilesTabView.swift`) и формулировок — с "выбранные приложения роутятся автоматически" на "приложения, которые поддерживают ручную настройку прокси".

Затронутые файлы при реализации любого из вариантов: `ConfigTransformer.swift`, `SubscriptionParser.swift`, `ProxyManager.swift`, `AppRoutingStore.swift`, `GeneralTabView.swift`, `ProfilesTabView.swift`.
