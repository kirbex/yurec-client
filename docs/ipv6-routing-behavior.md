# IPv6 и Happy Eyeballs в разных режимах VPN

## Контекст

Браузеры используют алгоритм **Happy Eyeballs** (RFC 8305): при наличии и IPv4, и IPv6 адресов у домена браузер запускает оба соединения почти одновременно и использует то, которое установится первым. Это важно понимать, потому что именно это поведение создаёт проблему в гибридном режиме TUN+SOCKS5.

---

## Режим 1: без VPN

```
Browser
  │
  ├─ DNS-запрос example.ru
  │     ISP DNS отвечает:
  │     A:    203.0.113.10      (IPv4)
  │     AAAA: 2001:db8::1      (IPv6)
  │
  ├─── Happy Eyeballs IPv6 ──→ OS network stack
  │                                  │
  │                           нет глобального IPv6 на интерфейсе
  │                           OS → ICMP "no route to host" (мгновенно, до handshake)
  │                                  ✗ отказ ДО установки соединения
  │
  └─── Happy Eyeballs IPv4 ──→ OS network stack → 203.0.113.10
                                                   RU IP ✅

Итог: работает. IPv6 падает мгновенно на уровне OS,
      браузер автоматически переходит на IPv4.
```

**Почему это важно:** OS возвращает ошибку `EHOSTUNREACH` или `ENETUNREACH` **до** того, как TCP-соединение установлено. Браузер обрабатывает это как "адрес недоступен" и корректно переключается на IPv4.

## Режим 2: TUN-режим (`route.final = "proxy"`)

Весь трафик проходит через VPN-сервер.

```
Browser
  │
  ├─ DNS-запрос example.ru
  │     172.19.0.1 (TUN) → sing-box DNS → 1.1.1.1 через VPN-тоннель
  │     A:    203.0.113.10
  │     AAAA: 2001:db8::1
  │
  ├─── Happy Eyeballs IPv6 ──→ TUN-интерфейс → sing-box
  │                                                │
  │                                         route: proxy
  │                                                │
  │                               [sniff] извлекает SNI из TLS ClientHello
  │                               цель: "example.ru" (домен, не IP)
  │                                                │
  │                                    VLESS-тоннель (TCP IPv4)
  │                                    ──→ vpn-server.example.com:443
  │                                                │
  │                                    Сервер получает запрос:
  │                                    "подключись к example.ru:443"
  │                                                │
  │                                    Сервер резолвит example.ru
  │                                                │
  │                                  ┌─────────────┴─────────────┐
  │                             есть IPv6                   нет IPv6
  │                                  │                           │
  │                            → 2001:db8::1 ✅          → 203.0.113.10 ✅
  │                              (сервер имеет              (fallback на IPv4,
  │                               IPv6-связность)            сервер сам решает)
  │
  └─── Happy Eyeballs IPv4 ──→ (аналогично, тот же путь через VPN)

Итог: работает. Но внешний IP = IP VPN-сервера (иностранный).
      Сервисы с geo-блокировкой (Ozon, Кинопоиск) могут заблокировать.
```

**Ключевой момент:** sing-box передаёт VPN-серверу **доменное имя** (благодаря sniff/SNI), а не IPv6-адрес. Сервер сам резолвит домен и сам выбирает IP согласно своей конфигурации. Если у сервера нет IPv6 — он возьмёт IPv4 без ошибок.

**Исключение:** если sniff не срабатывает (HTTP без TLS, нестандартный TLS, QUIC), sing-box передаёт серверу оригинальный IPv6-адрес. Если у VPN-сервера нет IPv6 — соединение упадёт.

## Режим 3: SOCKS5 гибридный (`route.final = "direct"`)

Браузер идёт напрямую, выбранные приложения через VPN.

```
Browser
  │
  ├─ DNS-запрос example.ru
  │     172.19.0.1 (TUN) → sing-box DNS → 1.1.1.1 через VPN-тоннель
  │     A:    203.0.113.10
  │     AAAA: 2001:db8::1
  │
  ├─── Happy Eyeballs IPv6 ──→ TUN-интерфейс → sing-box
  │                                                │
  │                                         route: direct
  │                                                │
  │                            ┌── sing-box принимает TCP handshake ЛОКАЛЬНО
  │                            │   SYN → SYN-ACK → ACK  (< 1ms)
  │                            │   Браузер считает: "соединение установлено!"
  │                            │   IPv6 выигрывает гонку Happy Eyeballs
  │                            │
  │                            └── sing-box пытается открыть соединение наружу:
  │                                direct outbound → 2001:db8::1:443
  │                                      │
  │                                нет глобального IPv6 на машине
  │                                      │
  │                                sing-box → TCP RST → browser
  │
  │         Браузер получает RST на "установленное" соединение
  │         → ERR_CONNECTION_RESET 🚫
  │         → Happy Eyeballs НЕ переключается на IPv4
  │           (RST трактуется как сброс сервером, а не как недоступность адреса)
  │
  └─── Happy Eyeballs IPv4 ──→ TUN → sing-box → direct → 203.0.113.10
                                                           RU IP — но уже
                                                           не запускается,
                                                           IPv6 "победила"

Итог: ERR_CONNECTION_RESET. Браузер не откатывается на IPv4.
```

### Почему IPv4 не спасает

Разница между двумя видами отказа:

| Ситуация | Ошибка | Поведение браузера |
|---|---|---|
| Нет IPv6 без VPN | `EHOSTUNREACH` до handshake | Откатывается на IPv4 |
| TUN + direct, нет IPv6 | TCP RST после handshake | **Не откатывается**, показывает ошибку |

TUN создаёт "ловушку": принимает handshake локально (поэтому IPv6 побеждает в гонке), а потом не может пробросить соединение наружу. К этому моменту IPv4-попытка браузером уже отброшена.

## Режим 3 с фиксом: SOCKS5 + `strategy: ipv4_only`

```
Browser
  │
  ├─ DNS-запрос example.ru (AAAA)
  │     sing-box DNS (strategy: ipv4_only)
  │     → AAAA-запрос блокируется / не возвращается
  │     → возвращается только A: 203.0.113.10
  │
  └─── только IPv4 ──→ TUN → sing-box → direct → 203.0.113.10
                                                   RU IP ✅

Итог: работает. Браузер никогда не узнаёт об IPv6-адресе.
```

## Автоматический выбор стратегии в клиенте

Вместо того чтобы всегда отключать IPv6, клиент проверяет наличие глобального IPv6-адреса на физических интерфейсах в момент старта VPN:

```
DNSHelper.hasGlobalIPv6()
  │
  ├── перебирает все сетевые интерфейсы (getifaddrs)
  ├── исключает: utun*, lo* (loopback и TUN-интерфейсы)
  ├── исключает: link-local адреса (fe80::/10)
  │
  ├── нашёл глобальный IPv6 → strategy не меняется (IPv6 работает)
  └── не нашёл             → strategy = "ipv4_only"
```

Это позволяет корректно работать как на сетях без IPv6 (большинство домашних ISP), так и на сетях с полноценным dual-stack (офисы, дата-центры, облачные среды).

## Краткая сводка по режимам

| Режим | Браузерный IP | IPv6 соединение | Проблема |
|---|---|---|---|
| Без VPN | RU (ISP) | OS отказывает мгновенно → IPv4 fallback | нет |
| TUN | IP VPN-сервера | Идёт через VPN-сервер, сервер сам решает | geo-блок RU-сайтов |
| SOCKS5 hybrid (без фикса) | RU (ISP) | TUN принимает handshake → RST | ERR_CONNECTION_RESET |
| SOCKS5 hybrid (с фиксом) | RU (ISP) | AAAA не возвращается, только IPv4 | нет |

---

## Затронутые домены

Проблема проявляется на dual-stack сайтах, где IPv6-запись есть, но соединение по ней падает:

- Yandex и все его сервисы (Кинопоиск, Почта, Музыка и др.)
- Google (Gmail, Maps, Docs и др.)
- Другие крупные CDN с dual-stack

Сайты только с A-записями (например, многие российские e-commerce платформы) проблемы не имеют — браузер никогда не пытается IPv6.
